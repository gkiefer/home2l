CRcEventDriver Class Reference

Description

Local driver using the event processor mechanism for the 'DriveValue()' functionality.

The class is useful for the Python API to allow writing simple applications (e.g. 'rules' scripts) without parallel threads and an event loop design.

  • 'rcsBusy': (default) 'rcsBusy' with the old value is reported now; the application must report a valid and new value later.

◆ DriveValue()

virtual void CRcEventDriver::DriveValue ( CResource *  rc,
CRcValueState *  vs 
)
protectedvirtual

Drive a new value; Will be called from CResource::DriveValue().

Propagate a new requested value to the device (e.g. to an actor). Afterwards, the new value and state pointed at by 'vs' will be reported automatically by the caller.

If there are no pending requests at all, this method is called with 'vs->State() == rcsUnkown'. Most drivers will do nothing then (e.g. the "signal" driver does so) and keep the previous value active. Some drivers (e.g. brownies/shades) may switch off their actors now.

In all other cases, state of '*vs' is preset with 'rcsValid'.

It is legal to modify '*vs' to reflect a new state or even a value different from the driven one. The method shall change it to an appropriate state ('rcsUnknown' or 'rcsBusy') as adequate. If the new value is not known yet, the state should be set to 'rcsBusy' now and a 'CRcResource::Report...()' method called later with a valid value.

If the state is set to 'rcsBusy', the busy state will be reported with the new value by default (as stored in '*vs'). To report 'rcsBusy' with the old value instead, call CRcValueState::SetToReportBusyOldVal() on '*vs'.

To report report nothing at all this time, call CRcValueState::SetToReportNothing() on '*vs'.

It is not allowed to call any method of the '*rc' object (e.g. 'Report' methods) from here, since the resource may be locked.

In most cases, just writing to the device is sufficient.
