89 // DEPRECATED COMMENT: Check and update (TBD)
90 //
91 // The class 'CPhone' represents a telephone and corresponds to a
92 // 'LinphoneCore' object. It contains hard-coded two "Call"
93 // objects (#0 and #1). This is to support the following scenarios:
94 //
95 // a) Simple calls: only call #0 is used, #1 remains inactive.
96 //
97 // b) Transfer calls: #0 is the new call (or none), #1 the paused one.
98 // Swapping between slots #0 and #1 happens on pausing/resuming.
99 //
100 // c) Camera preview for the door phone using a second call is presently
101 // unsupported.
102 // Current solution for door cam: Use primary call (no second call),
103 // accept with mic muted. Drawback: Visitor recognizes accept action.
104 //
105 // d) Machines with multiple logical phones (door/gate phone): Use multiple
106 // 'CPhone' objects?
107 //
108 // NOTE on the use of audio / camera devices by other program modules / threads
109 // (e.g. application-programmed ringing, music playback):
110 //
111 // 1. To avoid conflicts, other modules must ensure that during the invocation
112 // of any action ('Dial', 'AcceptCall', enabling auto-acceptance),
113 // such devices are closed by the other modules.
114 //
115 // 2. Devices may be used by the other modules only after a "device-permitting"
116 // state has been reported again via 'ReportState ()'. The "device-permitting"
117 // states are: 'psIdle', 'psRinging'.
118 //
119 // To avoid race conditions, the following contract applies:
120 // - After returning from 'CPhone::Dial ()', the state is already 'psDialing'
121 // (i.e. 'ReportState ()' is called from 'CPhone::Dial ()').
122 // - 'AcceptCall ()' is only called when in state 'psRinging'.
123 // - If 'SetIncomingCallAction (psInCall)' is called, the application does
124 // not use any device at all.
